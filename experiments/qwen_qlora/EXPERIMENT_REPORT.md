## 3. Loss 结果

| 评估点 | Epoch | 训练阶段 Loss | 验证 Loss |
|---|---:|---:|---:|
| Step 50 | 0.84 | 0.2050 | **3.0649** |
| Step 100 | 1.67 | 0.0088 | 3.7191 |
| Step 150 | 2.50 | 0.0022 | 3.9590 |
| Step 180 | 3.00 | 0.0021 | 3.9741 |

最佳检查点为 `checkpoint-50`。训练脚本启用了 `load_best_model_at_end`，因此最终 `adapter/` 保存的是 Step 50 的最佳 LoRA 权重，而不是 Step 180 的过拟合状态。

## 4. 结果分析

训练过程正常完成，梯度、显存和训练速度均无异常，QLoRA 在 8GB 显存设备上具有良好的可运行性。

训练 Loss 快速下降到接近 0，而验证 Loss 从 Step 50 开始持续上升，表明模型在当前数据上存在明显过拟合。主要原因是 1,200 条样本由 133 个独立问题通过指令模板扩展得到，同一答案在训练集中重复出现多次；训练问题与验证问题又按 `group_id` 完全隔离，因此模型很容易记忆训练答案，但对未见问题的泛化有限。

验证 Loss 较高不等于生成回答一定变差。开放式问答中存在多种正确表达，交叉熵会严格惩罚与参考答案措辞不同的输出，因此还需要通过测试集生成结果、自动指标和人工评分判断实际回答质量。

## 5. 下一步

1. 运行 `evaluate.py`，对比基座模型和最佳 LoRA Adapter 在独立测试问题上的回答。
2. 填写 `results/human_review.csv` 中的正确性、完整性、结构性和面试适用性评分。
3. 运行 `summarize_results.py` 生成最终质量汇总。
4. 若需要第二轮训练，优先增加真实独立问题，不再重复扩展同一答案；并将 Epoch 调为 1、学习率降为 5e-5～1e-4。

## 6. 微调前后生成评测

本轮在 14 个独立测试问题上进行成对生成。

| 指标 | 微调前 | 微调后 | 结果 |
|---|---:|---:|---:|
| 字符 F1 | 0.1759 | 0.3603 | 提升 104.8% |
| ROUGE-L | 0.1131 | 0.2596 | 提升 129.5% |
| 平均生成耗时 | 20.86 秒 | 5.31 秒 | 约 3.93 倍加速 |
| 平均回答长度 | 695.4 字符 | 105.4 字符 | 缩短约 84.8% |

微调模型在字符 F1 上有 13/14 题高于基座模型，在 ROUGE-L 上有 12/14 题高于基座模型。说明模型明显学到了训练数据中“短答案、分点回答、接近参考措辞”的风格。

不过，自动指标提升不能等同于技术正确性提升。人工抽查发现：

- 单元测试、重构、注解、职业动机等题目更简洁，更适合作为面试口头答案。
- Redis Cluster 回答混入“垂直分裂、布隆过滤器、随机 leader”等错误概念。
- `sleep()` / `wait()` 回答遗漏所属类、调用条件和唤醒机制，并给出错误示例。
- redo log、undo log 的职责描述存在混淆。
- MySQL 隔离级别回答错误地写成“默认读未提交”。
- Redis 线程模型回答过度简化，遗漏 Redis 6.0 前后网络 I/O 与命令执行线程模型的区别。
- Java 基本类型回答出现“每个字节对应 1 个 bit”等事实错误。

因此本轮实验的严谨结论是：

> QLoRA 微调显著改善了回答长度、结构和参考答案相似度，也降低了生成时间；但由于独立知识问题数量少、训练答案存在噪声和模板重复，技术事实正确性不稳定，暂不适合直接替代基座模型用于生产问答。

推荐将该 Adapter 用于验证“回答风格微调”的可行性。下一轮应先清洗答案并扩充独立问题，再使用较低学习率和不超过 1 Epoch 的配置训练；技术正确性应以人工评审或权威答案核验为主。
